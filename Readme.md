# Проект Zeta-Stress: Высокопроизводительный конвейер верификации нулей Дзета-функции Римана на 124 ядрах

## 1. Общее описание системы
Проект представляет собой специализированный вычислительный комплекс, написанный на языке **Haskell** и развернутый в изолированной операционной среде **NixOS**. Цель проекта — массовый расчет и верификация нетривиальных нулей дзета-функции Римана (через вещественную Z-функцию Харди) на сверхвысоких высотах ($T \ge 10^{12}$) с параллельным сохранением сырых бинарных данных на дисковый массив.

Программа утилизирует **124 вычислительных ядра CPU** и работает в жестких рамках **950 Гб оперативной памяти** (при общем объеме сервера 990 Гб).

---

## 2. Математические и архитектурные решения

### Метод Одлыжко–Шёнхаге через БПФ (FFT)
Вместо лобового расчета каждой точки, сложность которого растет как $\mathcal{O}(T^{1/2})$, алгоритм разбивает интервалы на крупные блоки. Математическая сумма аппроксимируется рядами Тейлора, а расчет сетки точек сводится к **Дискретному преобразованию Фурье (DFT)**. С помощью низкоуровневой Си-библиотеки **FFTW** вычисляется одно тяжелое БПФ-окно (размером `1 048 576` сэмплов с применением техники *Zero Padding* до ближайшей степени двойки), что мгновенно дает значения функции для всего подблока.

### Ликвидация ленивых задержек (Strict Parallelism)
Классические ленивые списки Haskell генерируются последовательно в один поток, что приводило к простою процессоров. В финальной версии вычисления переведены на строгие параллельные векторы `Data.Vector.Storable` и стратегию `withStrategy (parListChunk 16 rdeepseq)`. Задачи нарезаются на пулы и жестко распределяются планировщиком GHC RTS по всем 124 ядрам одновременно.

### Высокоскоростной бинарный I/O без участия GC
Чтобы Сборщик мусора (Garbage Collector) Haskell не тратил процессорное время на сканирование сотен гигабайт памяти, массивы найденных нулей (вещественные числа типа `Double`, 8 байт на число) сбрасываются из ОЗУ на диск напрямую через Си-указатели `V.unsafeWith` и системный вызов `hPutBuf`. Запись происходит монолитным блоком через буфер вывода в 64 МБ на максимальной скорости шины контроллера.

## 3. Отказоустойчивость и управление хранилищем

1. **Абсолютная изоляция (NixOS Flakes):** Проект собирается с помощью LLVM-бэкенда компилятора GHC (`-fllvm`) и флагов оптимизации под конкретную NUMA-архитектуру процессора (`-optc-march=native`).
2. **Аппаратное ускорение хранилища:** 4 пустующих SAS SSD накопителя корпоративного класса (Samsung PM1643a/PM1653) по 1.7 Тб объединены в программный массив **RAID-0** (`/dev/md0`) силами утилиты `mdadm`. Это увеличило скорость дисковых операций почти в 4 раза (~1.5–2 Гб/сек), полностью исключив задержки ввода-вывода.
3. **Автоматическая нарезка томов (Том = 900 Гб):** Haskell-код в реальном времени мониторит размер файлов в папке `/home/ilya/zeta-stress/data/`. Данные автоматически нарезаются на файлы `riemann_zeros_part1.bin`, `riemann_zeros_part2.bin` и т.д. Лимит в 900 Гб гарантирует, что любой из этих файлов без проблем поместится на домашние терабайтные жесткие диски при скачивании по `scp`/`rsync`.
4. **Предохранитель переполнения:** В код встроен жесткий лимит. Как только программа заполнит 9-й файл (суммарно ~8.1 Тб данных) и попытается открыть 10-й, она выдаст предупреждение и безопасно остановит Bash-цикл, предотвратив критический сбой всей ОС из-за нехватки места.

## 4. Методология использования полученных файлов для тестирования СХД

Полученные бинарные файлы представляют собой идеальный синтетический материал высокой плотности для стресс-тестирования, бенчмаркинга и валидации промышленных Систем Хранения Данных (СХД). 

Поскольку файлы состоят из идущих подряд 64-битных чисел с плавающей точкой высокой энтропии (хаотичные триллионные мнимые части нулей), они ведут себя как «белый шум» для контроллеров дисковых массивов.

### Способ 1. Тестирование реальной дедупликации и компрессии СХД
Современные СХД (особенно All-Flash массивы от NetApp, PureStorage, Dell EMC) используют встроенное аппаратное сжатие данных «на лету» (Inline Compression) и дедупликацию для экономии места.
* **Суть теста:** Загрузите файлы `riemann_zeros_partX.bin` на тестируемую СХД.
* **Что проверяем:** Так как нули дзета-функции локально хаотичны (модель GUE случайных матриц), они обладают колоссальной информационной энтропией. Они практически **не дедуплицируются** (0% дубликатов на уровне блоков) и сжимаются только специализированными дельта-алгоритмами. Этот тест покажет **худший сценарий (Worst-Case Scenario)** для СХД: насколько сильно просядет производительность процессоров контроллера СХД, когда они попытаются безуспешно сжать несжимаемый 900-гигабайтный поток «чистой математики».

### Способ 2. Стресс-тест последовательного чтения (Sequential Read) на пределе шины
Каждый файл весит 900 Гб, что гарантированно превышает объем RAM-кэша любого коммерческого контроллера СХД (который обычно составляет от 32 до 256 Гб).
* **Суть теста:** Запустите многопоточное чтение файла `riemann_zeros_part1.bin` с помощью утилиты `fio` или скрипта на Python:
  ```bash
  # Пример теста fio для эмуляции чтения 32 потоками без использования кэша ОС
  nix-shell -p fio --run "fio --name=seq_read_test --filename=data/riemann_zeros_part1.bin --rw=read --bs=1M --direct=1 --numjobs=32 --time_based --runtime=600 --group_reporting"
  ```
* **Что проверяем:** СХД будет вынуждена непрерывно читать данные напрямую со слоя Flash-памяти (SSD), полностью минуя кэш. Это позволит измерить чистую, честную пропускную способность (Throughput в Мб/с) контроллеров и стабильность задержек (Latency) при многочасовой максимальной нагрузке.

### Способ 3. Валидация алгоритмов предвыборки (Read-Ahead / Prefetch) СХД
Числа в бинарном файле идут строго по возрастанию аргумента $t$. Это создает идеальный паттерн строго последовательного доступа на уровне байтов.
* **Суть теста:** Проверяется скорость чтения небольшими блоками (например, по 4Кб или 8Кб), но строго последовательно.
* **Что проверяем:** Интеллектуальные контроллеры СХД должны мгновенно распознать этот паттерн и активировать алгоритм *Prefetch* (начнут заранее подгружать следующие блоки из SSD в кэш контроллера до того, как их запросит процессор). Если СХД умная — скорость чтения файлов с нулями будет близка к теоретическому максимуму. Если алгоритм предвыборки в прошивке СХД написан плохо — вы увидите сильные просадки и скачки задержек (latency spikes).

### Способ 4. Тестирование стабильности пропускной способности сети (SAN/NAS)
Перенос файлов размером 900 Гб по протоколам iSCSI, Fibre Channel, NFS или SMB на домашние хранилища — это отличный тест для сетевой инфраструктуры.
* **Суть теста:** Длительное копирование томов через `rsync` или `scp`.
* **Что проверяем:** Позволяет выявить скрытый перегрев сетевых карт (NIC), стабильность буферов коммутаторов и пропускную способность сетевых стыков при многочасовой непрерывной передаче монолитного неразрывного трафика Big Data.

# Построение
```bash
nix build
```

# Запуск
```bash
nix-shell -p tmux --run "tmux new-session -A -s zeta '
  START=1000000000000
  STEP=50000000
  BLOCK_NUM=1
  
  if [ ! -f riemann_chunks_summary.csv ]; then
    echo \"block_number;t_start;t_end;zeros_count\" > riemann_chunks_summary.csv
  fi

  while true; do
    END=\$((START + STEP))
    ./result/bin/zeta_stress \$BLOCK_NUM \$START \$END +RTS -N124 -M950G -A128m -RTS
    START=\$END
    BLOCK_NUM=\$((BLOCK_NUM + 1))
    sleep 0.1
  done
'"
```
